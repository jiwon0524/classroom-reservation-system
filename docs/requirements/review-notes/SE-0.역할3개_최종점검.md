# 역할 3개 체계 최종 점검

작성일: 2026-05-09

## 최종 역할

| 역할 | 설명 |
|---|---|
| 학생 | 개인 학습/조별 학습 목적으로 강의실 예약을 신청하고 본인 예약 조회/취소/변경 요청, 사용 후 시설 상태 확인을 수행한다. |
| 교수 | 보강, 세미나, 학생 지도 목적으로 강의실을 예약하고 본인 예약 조회/취소/변경 요청, 사용 후 시설 상태 확인을 수행한다. |
| 관리자 | 운영자 역할이다. 예약 승인/거부, 예약 변경 승인/거부, 대기 승격 확인, 시간표 관리, 강의실/시설 관리, 운영 공지, 백업/복구, 감사 로그, 시설 신고 처리를 담당한다. |

개발 환경/테스트/산출물 관련 항목은 로그인 역할이 아니라 `시스템` 품질 및 설계지침서 반영 항목으로 정리했다.

## 점검 결과

| 점검 항목 | 결과 |
|---|---|
| SRS 이해관계자 | 학생, 교수, 관리자, 시스템만 사용 |
| StRS 이해관계자 | 학생, 교수, 관리자, 시스템만 사용 |
| 권한표 컬럼 | 학생, 교수, 관리자 3개 역할만 존재 |
| 사용자 파일 role | STUDENT, PROFESSOR, ADMIN |
| 백업/복구 실행 권한 | 관리자 |
| 승인/거부 권한 | 관리자 |
| 예약 변경 승인/거부 권한 | 관리자 |
| 운영 공지 등록/수정 권한 | 관리자 |
| 시설 신고 처리 권한 | 관리자 |

## 최종 판단

역할 구조가 깔끔해졌다. 시스템 사용자는 `학생 / 교수 / 관리자` 3개만 가지며, 기술 제약과 품질 요구사항은 `시스템` 항목으로 분리되어 권한 모델과 설계지침서 항목이 섞이지 않는다.
