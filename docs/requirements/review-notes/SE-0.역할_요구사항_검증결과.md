# SE-0 역할/요구사항 검증 결과

검증일: 2026-05-09

## 검증 결론

최종 요구사항 산출물은 사용자 역할을 `학생`, `교수`, `관리자` 3개로 정리했으며, 운영 보조 역할은 `관리자` 권한으로 통합되어 있습니다. 구현 제약과 산출물 작성 책임은 로그인 역할이 아니라 `시스템` 품질 및 설계지침서 반영 항목으로 분리했습니다.

최종 개수는 StRS 36개, SRS 63개입니다. SRS는 기능 요구사항 47개와 비기능 요구사항 16개로 구성됩니다.

## 기준 고정 사항

- 체크인 창: 예약 시작 30분 전부터 예약 시작 10분 후까지, 표기 기준은 `-30분~+10분`
- 노쇼: 학생 예약은 시작 10분 후까지 미체크인 시 자동 취소 가능, 교수 예약은 관리자 판단 후보로 표시
- 층별 혼잡도: `(정규 강의 시간 + 승인 예약 시간 + 사용 불가 시간) / 전체 예약 가능 시간 * 100`
- 혼잡도 등급: 0~39 여유, 40~69 보통, 70~89 혼잡, 90~100 매우 혼잡
- 시설 신고 상태: RECEIVED, IN_PROGRESS, RESOLVED, REJECTED
- 예약 변경: 예약 시작 1시간 전까지만 요청 가능, 관리자 승인 시 예약 정책 재검사
- 대기 승격: 빈 시간이 생기면 같은 강의실/시간대 대기열에서 신청 시각이 빠른 예약을 우선 승격
- 사용 후 점검: 예약 종료 후 24시간 이내 기록 가능, 문제 선택 시 시설 신고로 연결
- 운영 공지: INFO/WARNING/BLOCKING으로 관리, BLOCKING은 신규 예약 차단
- 파일 저장: DB 없이 YAML/CSV 파일, 임시 파일 저장 후 검증 및 교체, 예약 단위 잠금, 백업 manifest 포함

## 마지막 점검 의견

6주 4인 프로젝트 기준으로 요구사항 문서는 충분히 풍부합니다. 다만 모든 기능을 동일한 완성도로 만들면 일정이 빡빡합니다. 1차 구현은 로그인/권한, 강의실 조회, 예약 신청/승인/취소, 파일 저장, 체크인/노쇼, 예약 변경, 시설 상태 표시까지 완성하고, 대기 승격/운영 공지/사용 후 점검/통계/백업은 발표 시나리오 중심으로 구현하는 것이 좋습니다.
