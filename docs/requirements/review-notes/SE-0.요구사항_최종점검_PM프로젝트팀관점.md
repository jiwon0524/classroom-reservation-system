# 강의실 예약 시스템 요구사항 최종 점검

작성일: 2026-05-09  
관점: 프로젝트팀 + PM + 시장/현업 서비스 비교

## 결론

현재 요구사항은 6주 4인 프로젝트 기준으로 방향이 좋고 발표용 매력도도 충분하다. 단순 예약에서 끝나지 않고 `정보관 1층~9층`, `콘센트/충전 가능 좌석`, `시설 상태`, `시설 불편 신고`, `층별 현황`, `체크인/노쇼 방지`, `공간 사용률`, `예약 변경`, `대기 예약 자동 승격`, `사용 후 시설 상태 확인`, `운영 공지`까지 들어가 있다.

## 잘 된 점

| 영역 | 판단 |
|---|---|
| 역할 체계 | 학생, 교수, 관리자 3개로 단순하고 명확함 |
| 원본 보존 | 기존 요구사항은 운영 역할만 관리자에 통합하고 기능 의도는 유지 |
| 설계지침서 반영 | DB 금지, 파일 저장, Java/Swing, Maven, JUnit5, UML/PlantUML, git 반영 |
| 시장성 | 공간 메타데이터, 체크인, 노쇼, 사용률, 시설 상태가 반영됨 |
| 구현 가능성 | MVP와 선택 기능을 나누면 6주 4인 프로젝트로 가능 |
| 운영 시나리오 | 예약 변경, 대기 승격, 사용 후 점검, 운영 공지까지 연결되어 실제 서비스처럼 보임 |

## 남은 주의점

1. 구현 시에는 체크인/노쇼, 예약 변경, 대기열 자동 승격을 한 번에 무리해서 만들지 않는다.
2. 파일 저장소는 temp 저장 후 교체 방식으로 먼저 안정화한다.
3. 권한 테스트는 학생/교수/관리자별로 꼭 작성한다.
4. 공간 사용률은 예약 기능이 안정된 뒤 발표용 통계 화면으로 구현한다.
5. 운영 공지는 `INFO/WARNING/BLOCKING` 3단계만 구현해 범위를 제한한다.
