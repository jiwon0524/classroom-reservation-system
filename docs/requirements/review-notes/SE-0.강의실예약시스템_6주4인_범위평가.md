# 강의실 예약 시스템 6주 4인 프로젝트 범위 평가

평가일: 2026-05-09

## 총평

현재 작성안은 요구사항 문서로는 좋다. 원본 요구사항을 보존했고, 설계지침서의 제한 조건도 반영했으며, 정보관 1층~9층 확장과 매력적인 기능도 들어가 있다. 최종 기준으로 StRS는 36개, SRS는 63개이며 기능 요구사항 47개와 비기능 요구사항 16개로 구성된다.

다만 4명이 6주 동안 Java Swing, 서버-클라이언트 구조, 파일 저장소, JUnit5 테스트, UML/PlantUML 산출물까지 모두 해야 한다면 전체 요구사항을 전부 구현하는 것은 범위가 크다. 성공하려면 기능을 `필수 구현`, `우선 구현`, `시연용 선택 기능`으로 나누어야 한다.

## 필수 MVP

| 기능 묶음 | 필수 구현 범위 |
|---|---|
| 계정/권한 | 학생, 교수, 관리자 로그인/로그아웃, 역할별 메뉴 |
| 강의실/시설 관리 | 정보관 1~9층 강의실 데이터 조회, 9층 지정 호실 초기 데이터 |
| 시간표/현황 조회 | 일별/주별/월별 현황, 정규 강의/예약 구분 표시 |
| 예약 신청 | 교수 최대 3시간, 학생 최대 2시간, 최소 하루 전 신청, 참석자 입력 |
| 예약 정책 | 시간 충돌 검사, 교수 예약 충돌 제한, 학생 정원 50% 제한, 동시성 검사 |
| 승인/취소 | 관리자 승인/거부, 본인 취소, 관리자 취소, 취소 사유 |
| 알림 | 승인/거부/취소 알림 저장, 로그인 시 표시 |
| 파일 저장 | YAML 또는 JSON 파일 저장, DB 사용 금지 |
| 백업/복구 | 강의실/시간표/예약 데이터 백업과 복구 |
| 테스트 | 예약 충돌, 정원 50%, 상태 전이, 권한 테스트 |

## 발표용 선택 기능

1. 층별 대시보드
2. 시설 필터
3. 콘센트/충전 가능 좌석 안내
4. 시설 불편 신고
5. 예약 변경 요청과 관리자 승인
6. 대기 예약 자동 승격
7. 사용 후 시설 상태 확인
8. 층/강의실 운영 공지

## 시간 남으면

1. 체크인/노쇼 방지
2. `.ics` 일정 내보내기
3. 감사 로그 조회 UI
4. 운영 공지의 기존 예약 영향 분석

결론적으로, 문서는 풍부하게 유지하되 구현은 MVP와 발표용 기능 중심으로 가져가는 것이 안전하다. 추가된 기능은 문서 매력도를 높이지만, 구현에서는 예약 변경과 대기 승격을 먼저 잡고 사용 후 점검과 운영 공지는 단순 화면 중심으로 구현하는 편이 좋다.
